# CODEBUDDY.md

This file provides guidance to CodeBuddy Code when working with code in this repository.

## Overview

This is a personal blog ("Eli Qiao's Blog") built on **Jekyll-Bootstrap (JB) v0.3.0** and deployed via **GitHub Pages** at `http://taget.github.io`. Content is written in Markdown; the site is generated by Jekyll. There is no separate build toolchain beyond Ruby/Jekyll — GitHub Pages compiles `_site/` automatically on push to the default branch.

## Common Commands

All authoring tasks go through the `Rakefile`:

```bash
# Local preview with live reload (serves at http://localhost:4000)
rake preview            # runs `jekyll serve -w`

# Create a new blog post in _posts/ (front matter is scaffolded automatically)
rake post title="A Title" [date="2014-06-09"] [tags=[tag1,tag2]] [category="linux"]

# Create a new top-level page
rake page name="about.html"        # or a path like name="docs/index.html"

# Switch the active theme (regenerates files in _layouts/)
rake switch_theme name="bootstrap-3"   # available: twitter, bootstrap-3

# Package / install JB themes (advanced, rarely needed)
rake theme:package name="bootstrap-3"
rake theme:install git="<public-clone-url>"
```

Raw Jekyll build (if needed): `jekyll build` writes to `_site/` (git-ignored). There are **no tests or linters** in this repository.

## Architecture

The layout follows the standard Jekyll-Bootstrap "JB" convention. The key indirection to understand is the **theme system**, which is why editing layouts is not as simple as editing `_layouts/`.

- **`_config.yml`** — Site + JB config. All JB-specific settings live under the `JB:` hash (comments provider, analytics, sharing, base paths). `BASE_PATH`/`ASSET_PATH` control URL prefixing for different deployment targets (CNAME vs. user page vs. project page).

- **Theme rendering chain** (most important to understand before editing markup):
  1. `_layouts/{default,post,page}.html` are **thin generated wrappers**. They contain only front matter + `{% include JB/setup %}` + `{% include themes/<theme>/<file> %}`.
  2. The real markup lives in **`_includes/themes/<theme>/`** (`twitter` and `bootstrap-3`). `bootstrap-3` is the active theme.
  3. `rake switch_theme` regenerates the `_layouts/*.html` wrappers to point at a different theme dir — **do not hand-edit `_layouts/` to change design; edit the theme includes or run switch_theme.**

- **`_includes/JB/`** — Framework helpers invoked from templates via `{% include JB/<name> %}`: `setup` (sets `BASE_PATH`/`ASSET_PATH`/`HOME_PATH` liquid vars used site-wide), `posts_collate`, `tags_list`, `categories_list`, `pages_list`, plus pluggable `comments`, `analytics`, and `sharing` dispatchers that read provider config from `_config.yml`.

- **`_posts/`** — Posts organized into subfolders by topic (`common/`, `linux/`, `openstack/`). Filenames must follow `YYYY-MM-DD-slug.md`. Every post starts with YAML front matter (`layout: post`, `title`, `category`, `tags`) followed by `{% include JB/setup %}`. Use `rake post` rather than creating files by hand to get the correct scaffold. Post body content is frequently written in Chinese.

- **`_drafts/`** — Unpublished drafts (not rendered in normal builds).

- **`assets/themes/<theme>/`** — Per-theme CSS/JS/fonts (Bootstrap). Referenced in templates via the `{{ ASSET_PATH }}` liquid variable, not hardcoded paths.

- **`_plugins/debug.rb`** — A custom `{{ var | debug }}` Liquid filter. Note: custom plugins do **not** run on GitHub Pages' default build (safe mode), so this only works in local `rake preview`.

- **Generated/index pages** — `archive.html`, `categories.html`, `tags.html`, `pages.html`, `atom.xml`, `rss.xml`, `sitemap.txt`, `index.md` use the JB helpers to list/aggregate posts.

## Key Conventions

- Always prefix internal links with `{{ BASE_PATH }}` (or `{{ HOME_PATH }}`) so URLs resolve correctly across deployment modes — never hardcode `/`.
- Reference theme assets through `{{ ASSET_PATH }}`.
- `README.md`, `Rakefile`, and `changelog.md` are excluded from the build via the `exclude` list in `_config.yml`.
- `_site/` is git-ignored and is build output — never edit it directly.
